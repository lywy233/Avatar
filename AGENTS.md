# AGENTS.md

## 工作规则（Working rules）
- 在进行任何修改之前，先阅读 `README.md` 以及 `docs/` 目录下的相关文档。
- 先简要说明修改计划，再进行代码更改。
- 保持修改**最小化且局部化**。
- 除非明确要求，不要修改任务范围之外的文件。
- 不要修改锁文件（如 `uv.lock`）、生成文件或构建产物，除非任务明确要求。
- 除非明确要求，不要更改 CI、Docker、部署配置或项目环境设置。

## Python / uv 约定（Python / uv conventions）
- 使用 `uv` 进行依赖管理和命令执行。
- 优先使用 `uv sync` 安装项目定义的依赖。
- 仅在明确要求时使用：
  - `uv add <package>` 添加依赖
  - `uv remove <package>` 移除依赖
- 优先使用 `uv run <command>` 执行：
  - 脚本
  - 测试
  - lint
  - Python 入口程序
- 在项目配置支持的情况下，优先使用 `uv` 管理的 Python 解释器。
- 不要切换到其他包管理工具（如 `pip`、`poetry`、`pipenv`），除非明确要求。
- 不要创建或修改与 `uv` 冲突的虚拟环境配置。

## 安全约束（Safety constraints）
- 在执行以下操作前必须先获得批准：
  - 删除文件
  - 重命名对外公开的 API
  - 修改数据库 schema（结构）
  - 引入破坏性变更（breaking changes）
- 不要执行具有破坏性的命令。
- 除非明确要求，不要安装新的依赖。
- 除非明确要求，不要批量升级依赖版本。
- 除非任务要求，不要进行大规模重构或重写模块。

## 代码风格（Code style）
- 遵循现有代码风格、结构和命名规范。
- 优先进行小范围修改，而不是大规模重构。
- 保持函数和类职责单一。
- 仅在逻辑不直观时添加注释。
- 除非明确允许，保持向后兼容。
- 使用 Google 风格 docstring 写注释，在由你新建函数/类的事情，请编写注释


## 测试与验证（Testing and validation）
- 优先以**最小范围**验证改动。
- 优先编写或运行针对性测试，而不是直接跑全量测试。
- 使用项目已有的测试和 lint 命令。
- 运行命令时优先使用：
  - `uv run pytest`
  - `uv run ruff check .`
  - `uv run python -m <module>`
- 如果未能执行验证，需要明确说明原因。

## 文件修改规则（File editing rules）
- 优先修改已有文件，除非确实需要新增文件。
- 除非明确要求，保持对外接口稳定。
- 避免修改与当前任务无关的文件。
- 不要手动修改生成文件，除非流程明确要求。

## 沟通规范（Communication）
- 修改前简要说明计划。
- 修改后说明做了哪些改动以及原因。
- 明确说明假设、限制或未验证的部分。
- 对潜在风险改动提前进行提示。
